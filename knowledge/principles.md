# 原則

## 根公理

**資訊的展示模態和儲存模態可以相異，但必須表示同樣的資訊，且轉換成本不能高於原有的成本。**

影片只是音檔、背景影像、文字字幕的一種「展示投影」。既然資訊本身可以用更精簡的形式儲存，就不需要預先合成成影片來佔用儲存空間。

## 衍生原則

### 1. 即時合成取代預存影片

推導自：根公理
意義：使用者請求觀看時才將素材合成為串流，不預先產生並儲存完整影片檔。
如何應用：系統收到播放請求時，讀取音檔、背景、字幕等素材，即時合成為影片串流輸出。

### 2. 轉換成本守恆

推導自：根公理（「轉換成本不能高於原有的成本」）
意義：即時合成的運算成本加上分離儲存成本，MUST 低於直接儲存完整影片的成本。若違反此條件，該場景應回退為預存影片。
如何應用：評估新功能時，MUST 比較合成方案與預存方案的總成本（儲存 + 運算 + 頻寬）。

### 3. 模組化媒體處理

推導自：根公理（展示與儲存模態分離）
意義：音檔、影像、文字為獨立的媒體類型，各自有獨立的處理邏輯與串流介面。
如何應用：每種媒體處理器 MUST 實作統一的串流介面，可獨立測試與替換。合成引擎透過介面組合，不直接相依內部實作。

### 4. 資訊等價性 MUST 可驗證

推導自：根公理（「必須表示同樣的資訊」需要可驗證的保證）
意義：合成結果與原始素材之間的資訊等價性（音訊時長、字幕對齊、分段總長、檔案完整性）MUST 有自動化驗證；等價性無關的輔助功能（UI toolbar、設定頁面、HTTP 端點外框）則以整合測試或手動驗收為主，不強制 Red-Green-Refactor。
如何應用：凡觸及合成管線、分段切割、playlist 產生、下載打包、快取清掃的程式碼，MUST 搭配自動化測試覆蓋等價性條件；前端互動與外掛 UI 可以在手動驗收後再補測試。

### 5. 簡約原則

推導自：原則 2（轉換成本守恆）
意義：過度設計會增加轉換成本，違反根公理。從最簡單可行的方案開始。
如何應用：MUST 遵循 YAGNI，不實作尚未確定的需求。新增抽象層 MUST 有至少兩個具體使用場景佐證。
