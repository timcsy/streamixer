# Feature Specification: HLS 即時合成串流

**Feature Branch**: `001-hls-stream-synthesis`
**Created**: 2026-04-04
**Status**: Draft
**Input**: 給定音檔、背景圖片、字幕檔，透過 HLS 即時合成為可播放的影片串流

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 播放合成影片串流 (Priority: P1)

教會網站的訪客點擊一則講道的播放按鈕，系統接收到播放請求後，將該則講道的音檔與背景圖片即時合成為影片，並以 HLS 串流的方式回傳。訪客在瀏覽器中看到的是一段帶有背景畫面和音訊的影片，體驗與播放一般影片無異。

**Why this priority**: 這是整個服務的核心價值——讓使用者能觀看「影片」，但伺服器只需儲存音檔和圖片。沒有這個功能，其他一切都沒有意義。

**Independent Test**: 提供一個音檔和一張背景圖片，向服務發送請求，用標準 HLS 播放器驗證能正常播放含音訊與畫面的影片串流。

**Acceptance Scenarios**:

1. **Given** 伺服器上有一個音檔和一張背景圖片，**When** 使用者請求該內容的 HLS playlist，**Then** 系統回傳有效的 .m3u8 檔案，其中包含正確的分段資訊
2. **Given** 使用者取得了 HLS playlist，**When** 播放器依序請求各 .ts 分段，**Then** 每個分段包含背景畫面與對應時段的音訊，可連續播放
3. **Given** 音檔長度為 30 分鐘，**When** 使用者從中間位置開始播放，**Then** 系統能提供對應時間點的分段，支援隨機存取

---

### User Story 2 - 顯示字幕 (Priority: P2)

訪客在觀看講道影片時，希望能看到同步的字幕文字。系統將字幕資訊整合到合成的影片串流中，讓文字與音訊同步顯示。

**Why this priority**: 字幕對於聽障者、非母語使用者或在無聲環境下觀看的人至關重要，但核心播放功能（音檔+圖片）不依賴字幕即可獨立運作。

**Independent Test**: 提供音檔、背景圖片和字幕檔，請求合成串流後，驗證播放器能顯示與音訊同步的字幕文字。

**Acceptance Scenarios**:

1. **Given** 伺服器上有音檔、背景圖片和字幕檔，**When** 使用者請求該內容的 HLS playlist，**Then** playlist 中包含字幕軌的相關資訊
2. **Given** 字幕檔中某段文字標記在 00:01:30 到 00:01:35，**When** 播放器播放到該時間點，**Then** 對應的字幕文字正確顯示
3. **Given** 沒有提供字幕檔，**When** 使用者請求播放，**Then** 系統正常播放影片串流，不顯示任何字幕

---

### User Story 3 - 容器化部署 (Priority: P3)

系統管理員希望能透過 Docker 快速部署此服務到現有的伺服器環境中，不需複雜的安裝與設定流程。

**Why this priority**: 部署的便利性影響服務能否被實際採用，但不影響核心合成功能的正確性。

**Independent Test**: 使用 Docker 指令啟動容器，驗證服務能正常接受請求並回傳合成的串流。

**Acceptance Scenarios**:

1. **Given** 已安裝 Docker 的伺服器，**When** 管理員執行容器啟動指令，**Then** 服務在 60 秒內可接受請求
2. **Given** 容器正在運行，**When** 管理員指定素材檔案的存放位置，**Then** 服務能存取並使用這些素材進行合成

---

### Edge Cases

- 音檔格式不受支援時，系統 MUST 回傳明確的錯誤訊息，說明支援的格式
- 背景圖片解析度過大或過小時，系統 MUST 將其縮放至合適的輸出解析度
- 字幕檔格式錯誤或編碼不正確時，系統 MUST 略過字幕並正常播放影片，同時記錄警告
- 音檔損毀或無法讀取時，系統 MUST 回傳錯誤而非產生無聲串流
- 多個使用者同時請求同一內容時，系統 MUST 能正常服務每個請求

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 接受音檔（至少支援 MP3、WAV 格式）作為音訊來源
- **FR-002**: 系統 MUST 接受圖片檔（至少支援 JPG、PNG 格式）作為背景畫面
- **FR-003**: 系統 MUST 接受字幕檔（至少支援 SRT、VTT 格式）作為文字來源
- **FR-004**: 系統 MUST 將音檔與背景圖片即時合成為影片串流
- **FR-005**: 系統 MUST 以 HLS 協定輸出合成的串流，提供有效的 .m3u8 playlist 與 .ts 分段
- **FR-006**: 系統 MUST 支援 HLS 的隨機存取（seek），使用者可從任意時間點開始播放
- **FR-007**: 系統 MUST 在字幕檔存在時將字幕整合至 HLS 串流中
- **FR-008**: 系統 MUST 在字幕檔不存在時仍能正常提供音訊加背景的影片串流
- **FR-009**: 系統 MUST 提供 Docker 映像檔，支援容器化部署
- **FR-010**: 系統 MUST 在素材格式不受支援或損毀時回傳明確的錯誤訊息

### Key Entities

- **素材組合（Media Composition）**: 一組待合成的素材，包含一個音檔來源、一個背景影像來源，以及選擇性的字幕來源
- **音檔（Audio）**: 音訊資料，具有格式、時間長度等屬性
- **背景影像（Background）**: 靜態圖片，具有格式、解析度等屬性
- **字幕（Subtitle）**: 帶有時間標記的文字資料，具有格式、編碼等屬性
- **HLS 串流（Stream）**: 合成後的輸出，包含 playlist（.m3u8）與分段（.ts）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者發送播放請求後，5 秒內收到第一個可播放的串流分段
- **SC-002**: 合成的影片串流在標準 HLS 播放器（如瀏覽器內建播放器）中能連續播放，無中斷或錯誤
- **SC-003**: 字幕與音訊的同步誤差不超過 500 毫秒
- **SC-004**: 儲存相同內容，素材分離儲存的空間佔用 MUST 小於完整影片檔的 50%

## Assumptions

- 使用者透過穩定的網路連線觀看串流
- 輸出影片解析度固定為單一預設值（如 720p），不支援多解析度自適應串流
- 每次合成只處理一個音檔配一張背景圖片，不支援多圖片輪播或影片背景
- 字幕為預先製作好的檔案，非即時語音辨識產生
- 服務不負責素材的上傳與管理，素材由外部系統提供
- 不包含使用者認證與授權機制，所有請求均可存取
- 不包含 WordPress 外掛整合，僅提供 HTTP 端點供外部呼叫
