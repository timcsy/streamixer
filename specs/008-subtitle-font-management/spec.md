# Feature Specification: 字幕字體管理

**Feature Branch**: `008-subtitle-font-management`
**Created**: 2026-04-19
**Status**: Draft
**Input**: 階段 9：管理員可全站管理字幕字體（系統內建 + 使用者上傳），每個素材組合可選自己的字體；未選則用全站預設。

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 管理員為教會網站上傳指定字體並設為全站預設 (Priority: P1)

教會想讓所有講道影片統一使用教會購買或授權的品牌字體。管理員從 Streamixer 設定頁上傳字體檔，設為全站預設，之後新合成的所有素材字幕都使用該字體。

**Why this priority**: 若沒有這個功能，所有字幕都只能使用系統 fallback 字體。這是整個階段 9 的最小可用情境——單是「能換字體」本身就交付了階段目標。

**Independent Test**: 管理員上傳 `Church.ttf`、設為預設；重新合成任一素材的第一個分段後，字幕視覺上以 `Church.ttf` 的字形呈現（與預設字體外觀不同）。

**Acceptance Scenarios**:

1. **Given** 設定頁尚未上傳任何使用者字體，**When** 管理員上傳合法 `.ttf` 字體檔，**Then** 字體列表立即出現新字體，並可被選為全站預設。
2. **Given** 全站預設為使用者上傳的字體，**When** 管理員觸發任一素材的重新合成，**Then** 新分段的字幕以該字體渲染。
3. **Given** 上傳的檔案副檔名是 `.ttf` 但內容不是合法字體，**When** 送出表單，**Then** 系統拒絕上傳並顯示可讀的錯誤訊息。

---

### User Story 2 — 訪客看到使用系統內建字體的字幕 (Priority: P1)

管理員尚未上傳任何字體時，訪客播放影片仍能看到清楚可讀的中文字幕（使用預裝的系統字體）。

**Why this priority**: 確保階段 9 落地後不會破壞既有功能——既有素材、既有播放行為都必須保持正常。

**Independent Test**: 全新部署、未設定任何字體、已有素材組合 → 播放影片，字幕以預裝字體呈現，無亂碼或方框。

**Acceptance Scenarios**:

1. **Given** 未設定全站預設字體，**When** 訪客播放素材，**Then** 字幕以系統內建 CJK 字體渲染。
2. **Given** 未設定全站預設字體，**When** 管理員在設定頁開啟字體列表，**Then** 至少看到一個系統內建字體（含 CJK 支援），並被標示為「系統內建」。

---

### User Story 3 — 單一素材組合覆寫全站預設字體 (Priority: P2)

管理員想讓某場特別聚會的影片字幕用不同字體，但不改動其他素材的行為。在該素材的編輯頁選擇另一個已上傳的字體，只有這個素材受影響。

**Why this priority**: 擴充情境，並非核心功能。多數教會站點都會統一用全站預設。

**Independent Test**: 兩個素材組合都已同步；A 改用字體 `Special.otf`、B 保持「使用全站預設」。重新合成後 A 的字幕視覺不同於 B。

**Acceptance Scenarios**:

1. **Given** 素材編輯頁的字體下拉預設為「使用全站預設」，**When** 管理員選擇特定字體並儲存，**Then** 該素材的後續合成使用選定字體。
2. **Given** 素材指定了某字體，**When** 管理員改選「使用全站預設」並儲存，**Then** 下次合成使用當前全站預設。

---

### User Story 4 — 管理員刪除不再需要的上傳字體 (Priority: P2)

管理員清理字體列表，只能刪除「使用者上傳」的字體，系統內建字體無刪除按鈕。刪除前系統應提示該字體仍被哪些素材使用。

**Why this priority**: 長期維運所需；不屬於首次可用範圍。

**Independent Test**: 上傳 `A.ttf` → 沒有素材使用它 → 刪除成功，列表不再出現。上傳 `B.ttf` → 某素材指定它 → 刪除動作前會列出使用它的素材，管理員可選擇繼續或取消。

**Acceptance Scenarios**:

1. **Given** 使用者上傳字體列表顯示 `A.ttf`，**When** 管理員按刪除，**Then** `A.ttf` 從列表與儲存空間移除，且後續上傳同名檔案不衝突。
2. **Given** 字體 `B.ttf` 被 2 個素材指定，**When** 管理員按刪除，**Then** 系統先顯示使用清單，經確認後才刪除；已指定該字體的素材在下次合成自動回退到全站預設。
3. **Given** 系統內建字體，**When** 管理員開啟字體管理頁，**Then** 該字體不顯示刪除按鈕。

---

### User Story 5 — 字體上傳的防呆與容量控制 (Priority: P3)

避免使用者不小心上傳非字體檔或撐爆儲存空間。

**Why this priority**: 安全／維運邊界；核心流程不需要也能運作。

**Independent Test**: 上傳 `fake.ttf`（實際為 txt）→ 被拒絕；嘗試上傳超過單檔上限的檔案 → 被拒絕；上傳達到總數上限後再上傳 → 被拒絕。

**Acceptance Scenarios**:

1. **Given** 上傳表單，**When** 選擇單檔大小超過系統限制，**Then** 上傳被拒絕並說明原因。
2. **Given** 字體總數已達系統上限，**When** 再嘗試上傳，**Then** 系統拒絕並建議先刪除不用的字體。
3. **Given** 伺服器收到副檔名 `.ttf` 但檔頭不符字體規格的請求，**When** 驗證階段，**Then** 上傳被拒絕，儲存空間不留垃圾檔。

---

### Edge Cases

- **字體名稱重複**：使用者上傳的字體 family name 與系統內建同名時，需以來源標示避免管理員誤認；合成時需明確指定使用哪一個來源。
- **字體缺少 CJK 字符**：管理員指定的字體不含中文字符時，fallback 機制會以系統字體補齊。系統不應阻擋但需在設定頁提示「此字體可能不含中文字符」。
- **字體被刪除但素材仍指定**：合成時回退到全站預設，並在 sync 狀態顯示警告。
- **管理員短時間連續改全站預設**：每次改都影響後續合成；不影響已快取分段，只影響新產生的分段。
- **儲存空間滿**：上傳失敗時回傳明確錯誤，不半寫入。
- **字體檔含多個 family（`.ttc`）**：需能讓管理員選擇要使用其中哪個 family name；或預設使用第一個。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**：系統 MUST 提供「字體列表」功能，同時列出系統內建字體與使用者上傳字體，每筆標示來源、family name、檔案大小。
- **FR-002**：系統 MUST 支援管理員上傳 `.ttf`、`.otf`、`.ttc` 格式的字體檔。
- **FR-003**：系統 MUST 在上傳階段驗證檔案確實為合法字體（以檔頭識別，非僅看副檔名），不合法則拒絕並清理臨時檔。
- **FR-004**：系統 MUST 提供「刪除」能力，僅作用於使用者上傳字體；系統內建字體不可被刪除。
- **FR-005**：刪除使用者字體前，系統 MUST 先列出目前指定該字體的素材組合，經管理員確認後才真正刪除。
- **FR-006**：系統 MUST 讓管理員設定「全站預設字體」，並可從字體列表中任選一項。
- **FR-007**：每個素材組合 MUST 能選擇自己的字體，選項包含「使用全站預設」與所有可用字體。
- **FR-008**：合成字幕時，系統 MUST 以素材指定字體為先；若未指定，使用全站預設；若全站預設未設定或指定字體已不存在，使用系統內建 CJK 字體。
- **FR-009**：系統 MUST 限制單檔字體大小與字體總數上限，超過時拒絕上傳並清楚告知原因。
- **FR-010**：系統 MUST 持久儲存使用者上傳字體，容器重啟後字體仍可用。
- **FR-011**：系統 MUST 在字體刪除後，讓仍指定該字體的素材於下次合成自動回退至全站預設，不能因找不到字體而合成失敗。
- **FR-012**：字體列表 MUST 可由 WordPress 設定頁即時讀取，上傳/刪除動作完成後 UI 狀態同步更新。

### Key Entities

- **Font（字體）**：代表一個可用字體，屬性包含來源（系統內建 / 使用者上傳）、family name、檔案大小、副檔名、是否可刪。
- **Font Selection（字體指定）**：素材組合對字體的引用。屬性包含對應字體 family name 與「使用全站預設」兩種狀態。
- **Global Default Font（全站預設字體）**：單一設定值，引用某個 family name；未設定時代表使用系統內建 fallback。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**：管理員從按下「選擇檔案」到字體出現於列表的時間 ≤ 5 秒（檔案 ≤ 10MB 情境）。
- **SC-002**：上傳偽裝字體檔（非字體內容 + `.ttf` 副檔名）時，系統 100% 拒絕並回傳可讀錯誤訊息。
- **SC-003**：部署後首次使用、未上傳任何字體時，所有既有素材字幕仍正確顯示中文，無亂碼率 100%。
- **SC-004**：素材組合更換字體設定後，下次合成的字幕視覺與舊字體可區分（以肉眼可辨為判準）。
- **SC-005**：刪除使用者字體且有 ≥ 1 素材仍指定該字體時，管理員獲得警告並能預覽受影響清單的時間 ≤ 3 秒。
- **SC-006**：容器重啟後，使用者上傳的字體 100% 保留，不遺失。

## Assumptions

- 字體僅用於伺服器端合成字幕，不向瀏覽器分發；無需考慮 WOFF、字體子集化、跨來源字體載入。
- 字體管理為系統單一全域空間，不以使用者／角色分隔（所有 WP 管理員共用同一字體集）。
- 系統內建字體至少包含一款支援中文的 sans-serif。
- 字體 family name 為唯一識別；若有衝突（例如上傳與內建同名），以來源區分，合成時優先使用使用者上傳版本。
- 單檔大小上限預設 10 MB，總數上限預設 50 個；可透過後端設定調整（本階段不做 UI）。
- 刪除後的快取分段不立即失效；仍以原字體播放至其分段從快取被清除。
- 上傳與刪除端點沿用既有 API Key 保護機制；列表端點公開可讀（回傳字體 metadata，不包含檔案下載）。
