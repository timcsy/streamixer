# Feature Specification: 音檔與逐字稿下載

**Feature Branch**: `007-audio-transcript-download`
**Created**: 2026-04-18
**Status**: Draft
**Input**: 階段 8：讓訪客從素材組合的播放頁面下載原始音檔與管理員上傳的逐字稿全文檔，並與現有影片下載維持一致的生命週期（上傳、同步、清除）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 訪客下載原始音檔 (Priority: P1)

教會會友／訪客在講道素材的播放頁，除了可以下載合成影片外，還能直接下載講道的原始音檔，方便離線聆聽、上傳到 podcast 平台或分享給未能出席的家人。

**Why this priority**: 原始音檔是素材組合中最核心的內容載體；沒有影像和字幕的裝置（純音訊 podcast app、行車播放）也能使用。提供音檔下載等於把「一份素材、多種模態」的承諾兌現給真正的訪客。

**Independent Test**: 管理員上傳一個已發布的素材組合，訪客於前台播放頁點「下載音檔」，可在本機取得與原始上傳完全相同的音檔（雜湊一致）並能正常播放。

**Acceptance Scenarios**:

1. **Given** 一個音檔已同步的素材組合，**When** 訪客在播放頁點「下載音檔」，**Then** 瀏覽器下載一個可播放的音檔，副檔名與原始上傳相同（例如 `.mp3`、`.wav`、`.m4a`）。
2. **Given** 素材組合尚未同步或音檔缺失，**When** 訪客點「下載音檔」，**Then** 系統顯示友善錯誤訊息，不產生損壞或空檔。
3. **Given** 素材組合的中文 slug 含有多位元組字元，**When** 訪客點「下載音檔」，**Then** URL 能正確解析並取得檔案，無雙重編碼導致 404。

---

### User Story 2 — 訪客下載逐字稿 (Priority: P1)

訪客可從播放頁下載管理員上傳的單一份逐字稿全文檔（txt/pdf/doc/docx/md），用於閱讀、研經、列印、翻譯。

**Why this priority**: 逐字稿是聽覺障礙者與閱讀偏好者的主要管道，屬於無障礙需求；同時也是教會資料保存（archives）的重要補充。

**Independent Test**: 管理員在素材編輯頁上傳一份逐字稿檔案並同步，訪客在播放頁點「下載逐字稿」，取得與上傳檔案完全一致的內容並以原始格式開啟。

**Acceptance Scenarios**:

1. **Given** 管理員已上傳並同步逐字稿，**When** 訪客在播放頁點「下載逐字稿」，**Then** 取得完整檔案，副檔名保留上傳時的格式。
2. **Given** 素材組合沒有逐字稿，**When** 管理員開啟播放頁，**Then** 「下載逐字稿」按鈕不顯示或為停用狀態，不顯示壞連結。
3. **Given** 逐字稿為 TXT/MD 純文字檔，**When** 訪客點下載，**Then** 檔案以附件形式下載而非瀏覽器內嵌顯示。

---

### User Story 3 — 管理員在素材編輯頁管理逐字稿 (Priority: P2)

管理員在 streamixer CPT 的素材編輯頁有一個「逐字稿」欄位，可上傳、替換、清除單一份逐字稿檔案。儲存素材時逐字稿一併同步到 Streamixer 後端。

**Why this priority**: 是 P1 下載情境的前置條件，但單獨實作不會立即對訪客產生價值，故排在 P2。

**Independent Test**: 管理員新增或編輯素材組合、上傳逐字稿、儲存；到 Streamixer 後端可見 `transcript.<ext>` 檔案；清除逐字稿欄位並再次儲存後，後端檔案亦被移除。

**Acceptance Scenarios**:

1. **Given** 素材編輯頁，**When** 管理員上傳 `.pdf` 逐字稿並儲存，**Then** Streamixer 後端對應目錄出現 `transcript.pdf`。
2. **Given** 已同步的逐字稿，**When** 管理員替換成不同副檔名的檔案並儲存，**Then** 舊檔被覆蓋，新副檔名正確反映在下載 URL。
3. **Given** 已同步的逐字稿，**When** 管理員清除欄位並儲存，**Then** 下次同步移除後端的 `transcript.*`，且播放頁的下載按鈕消失。

---

### User Story 4 — 同步後自動清除本地逐字稿檔案 (Priority: P2)

當自動清除功能開啟時，逐字稿檔案與音檔、背景、字幕一樣，同步成功後從 WordPress 媒體庫移除，節省網站空間；但保留檔名供編輯頁顯示。

**Why this priority**: 維持既有生命週期一致性，避免逐字稿成為遺留在 WP 媒體庫的孤兒附件。

**Independent Test**: 啟用自動清除選項、上傳並同步逐字稿；確認 WP 媒體庫的 attachment 記錄與實體檔案皆被移除，但 Streamixer 後端的 `transcript.*` 保留；編輯頁仍能顯示「<原檔名>（已同步至 Streamixer，本地已清除）」。

**Acceptance Scenarios**:

1. **Given** 自動清除啟用且逐字稿已同步，**When** 同步成功，**Then** WP attachment 被刪除、post meta attachment id 歸零、原檔名保留於 `_filename` meta。
2. **Given** 本地已清除的逐字稿，**When** 訪客點「下載逐字稿」，**Then** Streamixer 後端仍可回傳檔案，不受本地清除影響。

---

### User Story 5 — 後台批次匯出音檔或逐字稿 (Priority: P3)

管理員在 streamixer CPT 列表頁可批次選擇多個素材組合，從下拉選單選「匯出音檔」或「匯出逐字稿」，瀏覽器依序觸發下載每一個已同步項目的對應檔案。

**Why this priority**: 對單一素材的日常維運影響不大，屬於長尾資料備份／移轉場景。

**Independent Test**: 選取 3 個已同步的素材組合、下拉選「匯出音檔」、按套用；瀏覽器依序觸發 3 個音檔下載（錯開避免被擋）。

**Acceptance Scenarios**:

1. **Given** 已同步的素材組合，**When** 管理員批次匯出音檔，**Then** 瀏覽器依序下載每個素材的音檔原檔。
2. **Given** 素材有逐字稿與音檔，**When** 管理員選「匯出逐字稿」，**Then** 只下載逐字稿，音檔不下載。
3. **Given** 選取項目中有尚未同步者，**When** 批次匯出，**Then** 系統跳過未同步項目並在通知中回報已匯出與跳過的數量。

---

### Edge Cases

- 訪客按下載鈕當下，素材組合被管理員刪除或改為草稿：系統回傳 404 並在前端顯示友善訊息，不產生空白檔案。
- 音檔副檔名罕見（如 `.opus`、`.flac`）：下載應保留原副檔名且 MIME type 正確，或至少以 `application/octet-stream` 觸發下載。
- 逐字稿檔案極大（如 50MB 掃描 PDF）：下載必須以串流方式送出，不可將整個檔案讀入記憶體。
- 逐字稿上傳成功但同步暫時失敗（網路中斷）：編輯頁顯示同步錯誤，訪客端的下載按鈕不顯示；重新儲存可再次嘗試同步。
- 同一素材組合同時有影片、音檔、逐字稿三個下載選項時，播放頁 toolbar 在窄螢幕（手機）仍能清楚呈現，不互相重疊。
- 逐字稿檔名含中文或特殊字元：`Content-Disposition` 須使用 RFC 5987 `filename*=UTF-8''` 形式，避免亂碼。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**：系統 MUST 提供一個公開端點，能依素材組合識別碼回傳原始音檔，副檔名與上傳時相同。
- **FR-002**：系統 MUST 提供一個公開端點，能依素材組合識別碼回傳逐字稿檔案，副檔名保留上傳時的格式。
- **FR-003**：WordPress 素材編輯頁 MUST 提供「逐字稿」欄位，支援 txt/pdf/doc/docx/md 格式，單檔上傳、替換、清除。
- **FR-004**：WordPress 同步流程 MUST 在上傳音檔、背景、字幕時，額外一併上傳逐字稿（若有）。
- **FR-005**：系統 MUST 在同步成功後，依使用者的「同步後自動清除本地檔案」設定，將逐字稿連同音檔、背景、字幕一起從 WordPress 端清除。
- **FR-006**：播放頁 MUST 顯示三顆下載按鈕（影片／音檔／逐字稿）；逐字稿按鈕僅在該素材有逐字稿時出現。
- **FR-007**：音檔與逐字稿下載 MUST 以附件形式觸發瀏覽器下載，而非內嵌顯示。
- **FR-008**：音檔與逐字稿的下載 URL MUST 能正確處理中文 slug，避免雙重編碼。
- **FR-009**：系統 MUST 在素材組合不存在或對應檔案缺失時回傳可辨識的錯誤（HTTP 404 或等同），不產生空白或損壞檔案。
- **FR-010**：WordPress 列表頁的批次操作 MUST 提供「匯出音檔」與「匯出逐字稿」選項，跳過未同步項目。
- **FR-011**：下載端點 MUST 以串流方式輸出檔案，支援長檔案不爆記憶體。
- **FR-012**：逐字稿與音檔檔名含中文或特殊字元時，下載標頭 MUST 使用符合 RFC 5987 的格式避免亂碼。

### Key Entities

- **素材組合（Streamixer Composition）**：由管理員建立、以可讀 slug 識別的一組媒體；已包含音檔、背景、字幕、影片產物，本功能新增逐字稿屬性。
- **逐字稿（Transcript）**：單一檔案附件，屬於某素材組合，僅一份；屬性包含檔名、副檔名、原始大小；儲存在 Streamixer 後端的素材目錄內。
- **原始音檔（Source Audio）**：素材組合建立時上傳的音檔；既有屬性，本功能新增「可直接下載」語意。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**：在具備已同步素材組合的環境下，訪客從播放頁點下載音檔到取得檔案的時間 ≤ 3 秒（區網內或音檔 ≤ 20MB 情境）。
- **SC-002**：長音檔（≥ 1 小時、≥ 100MB）下載期間服務記憶體使用量不隨檔案大小線性成長（以串流方式輸出）。
- **SC-003**：訪客下載的音檔與逐字稿與管理員上傳的原檔位元完全一致（以雜湊比對驗證）。
- **SC-004**：啟用自動清除且已同步的素材組合中，WordPress 媒體庫不殘留任何本功能範圍內的檔案（音檔、背景、字幕、逐字稿）。
- **SC-005**：中文 slug 的素材組合其音檔與逐字稿下載 URL 可在 Chrome、Safari、Firefox 三大瀏覽器直接開啟無 404。
- **SC-006**：訪客在播放頁完成一次「下載逐字稿」操作所需的點擊數 ≤ 1 次（單顆按鈕即觸發）。

## Assumptions

- 「逐字稿」為單一檔案；需要多檔支援時以未來階段處理。
- 音檔格式沿用既有上傳驗證規則（mp3/wav/m4a 等），本功能不擴充可接受格式。
- 逐字稿接受格式為 txt/pdf/doc/docx/md；其他格式（如 rtf、odt、zip）暫不支援。
- 附件儲存位置沿用現有 `/media/{id}/` 目錄結構，與音檔、背景、字幕同層。
- WordPress 已套用 CPT 強制傳統編輯器設定（階段 7.5 已完成），逐字稿 meta 欄位沿用相同 `$_POST` 儲存流程。
- 下載端點沿用現行 CORS 與 API Key 設定：讀取端點公開、上傳端點受保護。
- 前端下載按鈕的 UI 在手機版仍使用水平 toolbar；若空間不足則自動換行，不強制做 responsive 隱藏。
- 批次匯出沿用既有「依序開新分頁」的瀏覽器觸發機制，不做伺服器端打包 zip。
